# dtool Studio 概念手册

> 面向：Agent（助手一号、助手二号、Manager）+ Heye
> 版本：v0.2 | 2026-06-26
> 更新说明：统一为"模块"单一概念体系

---

## 一句话说清

**dtool Studio 是一个可视化的数据处理工具。** 你把模块放到工作区，填参数，点运行，引擎执行后结果推回来。

---

## 核心概念

### ① 模块（Module）—— 唯一的概念

**模块是 dtool Studio 里唯一的概念。** 一个模块有名字、有参数、能干一件事。

```
┌─────────────────────┐
│ 模块                  │
│ ┌─── 参数 ──────────┐│
│ │ 输入文本: [____]  ││
│ │ 编码方式: [▾    ] ││
│ └───────────────────┘│
│ [▶ 运行]  [↕ 展开]   │
└─────────────────────┘
```

**模块只有两种状态，但不影响使用：**

- **简单模块** — 内部是原子功能，不可再分（如 `base64_encode`）
- **复合模块** — 内部包含其他模块（如"URL 批量编码"内部有循环和编码子模块）

↓ **最大的设计点**：从外面看，简单模块和复合模块长得一样——都有名字、参数、运行按钮。**用户不需要区分。**

**"双击"或"展开"才揭示内部**。不展开，就不知道也不需要知道。

**这个设计意味着什么：**
- 官方提供的模块 = 系统写的复合模块
- 用户自己做的流程 = 自定义复合模块
- 加载一个"模板" = 预填充一个复合模块的内容
- **没有"模板"这个概念，只有"已经填好了内部结构的模块"**

### ② 循环（Loop）

一种特殊模块，内部包含子模块（substeps），重复执行指定次数。

```
用户看到：   [循环模块] 一张卡片，带展开箭头
展开后看到： [子模块 A] [子模块 B]（和外层一样的卡片排列）
执行方式：   子模块 A → 子模块 B → 输出 → 再重复
输出：       最后一次迭代的最后一步结果
```

循环受两层保护：模块层上限 ≤ 10000，引擎层有环境变量兜底。

### ③ 条件（Branch）

一种特殊模块，根据条件选择执行哪条分支（暂不可用，预留位置）。

### ④ 执行引擎（Engine）

运行在后端 Node.js 进程中的执行器，和前端通过 WebSocket 通信。

```
前端                        后端引擎
┌──────────────┐         ┌──────────────┐
│ 排列模块卡片   │ ──WS→ │ 解析模块定义  │
│ 填参数        │        │ 拓扑排序      │
│ 点运行        │        │ Worker 执行   │
│ 看实时结果    │ ←WS── │ 每步推结果    │
│ 点停止        │ ──WS→ │ Terminate     │
└──────────────┘         └──────────────┘
```

引擎特性：worker_threads 隔离执行、实时结果推送、可中断。

## 入门流程

```
① 打开 dtool Studio
   ── 看到一个空白模块（默认预填一个最简单的"输入→输出"示例）
   
② 从右侧面板选择需要添加的模块
   ── 侧边栏展示所有可用模块（官方 + 自定义，按分类排列）

③ 填入参数，点击运行
   ── 后端引擎开始执行，结果实时推回

④ 想深入改造 → 双击模块展开内部
   ── 看到内部子模块列表，可以增删改
   ── 改完后折叠回去，模块看外面和之前一样

⑤ 保存为自定义模块 → 下次在侧边栏"我的模块"里直接使用
```

```
┌────────────────────────────────────────────┐
│  用户界面（前端 SPA）                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ 侧边栏   │  │ 工作区   │  │ 运行结果 │  │
│  │ 模块库   │  │ 模块卡片 │  │ 实时日志 │  │
│  │ 分类浏览 │  │ 循环展开 │  │ 参数表单 │  │
│  └──────────┘  └──────────┘  └──────────┘  │
├────────────────────────────────────────────┤
│  WebSocket 通信层                           │
├────────────────────────────────────────────┤
│  后端执行引擎                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ 模块解析 │  │ 调度器   │  │ Worker池 │  │
│  │ Schema   │→│ 拓扑排序 │→│ 隔离执行 │  │
│  │ 校验     │  │ 流式推送 │  │ 资源限制 │  │
│  └──────────┘  └──────────┘  └──────────┘  │
├────────────────────────────────────────────┤
│  持久化层                                   │
│  SQLite（模块 + 执行历史）                  │
└────────────────────────────────────────────┘

部署方式: Docker Compose (studio-frontend + studio-backend)
外部依赖: 无（执行引擎不依赖 AI API）
```

## 命名体系

| 用在哪里 | 叫什么 | 说明 |
|---------|-------|------|
| 日常工作区 | **模块**（Module） | 唯一的核心概念，简单/复合都一样 |
| 画布上的一张卡片 | **模块**（也称步骤 Step） | "我有 3 个模块在工作区" |
| 循环 | **循环**（Loop） | 一种特殊模块，内部可放子模块 |
| 条件 | **条件**（Branch） | 一种特殊模块，暂不可用 |
| 执行器 | **引擎**（Engine） | 运行在后端的执行器 |

**没有"模板"这个词。** 模块就是模块。系统提供的模块叫"系统模块"，用户自己做的叫"自定义模块"，加载预置内容叫"预填充模块"。

## 数据流

```
模块参数（用户填）
    │
    ▼
  $param.xxx 引用  ────────→  模块内部步骤的 config 中
    │
    ▼
步骤执行（顺序 = steps 列表顺序）
    │
    ▼
  $steps.步骤ID 引用 ←─────── 上一步的输出被命名
    │
    ▼
loop 内部：子步骤按相同规则执行
    │
    ▼
最终输出 → 展示给用户
```

### 变量引用语法

| 写法 | 含义 | 示例 |
|------|------|------|
| `$param.input_text` | 引用用户填的参数 | config.text: "$param.input_text" |
| `$steps.read_input` | 引用某步骤的输出 | config.source: "$steps.read_input" |
| `"prefix-$param.x-suffix"` | 内联拼接 | config.label: "处理-$param.name-完成" |
| `$$` | 字面量美元符号 | config.price: "$$19.99" |

**不支持：** 表达式求值、运算拼接、函数调用。变量引用只做字典查找。

## 安全边界

| 边界 | 说明 |
|------|------|
| 模块定义 | 声明式 YAML，不含可执行代码 |
| 模块白名单 | 动态加载的模块需在注册表中 |
| Worker 隔离 | 每步在独立 worker_threads 中执行 |
| 循环保护 | 模块层上限 ≤ 10000 + 引擎环境变量兜底 |
| 数据不持久 | 执行结果存 SQLite（可选），不在外部传输 |

## 与旧 dtool 的关系

**旧 dtool（公共项目）**
- 纯前端 SPA，浏览器内执行
- 面向公网用户
- 进入长期维护模式，不做新功能

**dtool Studio（新项目）**
- 前端 + 后端架构
- 仅限个人/团队内部使用
- 活跃开发中

**双向反哺：** Studio 开发的纯函数模块可以移植到旧 dtool。UI 和交互理念也可以参考。引擎、架构层面的能力不能反哺（架构不同）。
