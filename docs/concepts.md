# dtool Studio 概念手册

> 面向：Agent（助手一号、助手二号、Manager）+ Heye
> 版本：v0.1 | 2026-06-26

---

## 一句话说清

**dtool Studio 是一个可视化的数据处理编排工具。** 你安排步骤，后端引擎执行，结果实时推回给你看。

---

## 核心概念

### ① 步骤（Step）

最小的功能单元。每个步骤做一件具体的事：
- `base64_encode` → 把输入变成 Base64
- `combine` → 把两段数据拼起来
- `output` → 把结果显示出来

**用户看到的步骤 = 一张卡片**，从上到下排列在画布上。

步骤可以命名（通过 `export` 字段），命名后的输出可以被其他步骤引用：`$steps.步骤ID`。

### ② 模板（Template）

一个**完整的、可直接用的数据处理流程**。用户看到的模板是参数表单：
- 输入框（填数据）
- 下拉选择（选模式）
- 数字输入（调次数）
- 点击「运行」→ 引擎执行 → 看到结果

模板不需要用户理解内部结构。想深入改造时可以点"查看步骤"展开内部。

**模板 ≠ 子流程。** 模板是工具，子流程是技术概念。

### ③ 子步骤（Substep）

循环（loop）内部的步骤集合。子步骤和外层步骤的**结构完全一致**（也是卡片列表），区别在于：
- 子步骤在循环作用域内执行
- 可以引用外层的步骤输出
- 外层的步骤不能看到循环内部

### ④ 循环（Loop）

一种特殊步骤，内部包含一组子步骤，重复执行指定次数。

```
用户看到：    [循环步骤] 卡片，带一个展开箭头
展开后看到：  [子步骤1] [子步骤2] (和外层一样的卡片排列)
执行时：     子步骤1→子步骤2 重复 N 次
输出：       最后一次迭代的最后一步结果
```

循环受两层保护：模板层上限 ≤ 10000，引擎层有环境变量兜底。

### ⑤ 执行引擎（Engine）

运行在后端，和前端通过 WebSocket 通信。

```
用户编排（前端）         执行引擎（后端）
┌─────────────┐        ┌──────────────┐
│ 排列步骤卡片  │ ──WS→ │ 解析模板     │
│ 填参数       │        │ 拓扑排序     │
│ 点运行       │        │ Worker 执行  │
│ 看实时结果   │ ←WS── │ 每步推结果   │
│ 点停止       │ ──WS→ │ Terminate    │
└─────────────┘        └──────────────┘
```

引擎特性：worker_threads 沙箱隔离、实时结果推送、可中断执行。

## 架构概览

```
┌────────────────────────────────────────────┐
│  用户界面（前端 SPA）                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ 模板库   │  │ 步骤列表 │  │ 运行结果 │  │
│  │ 参数表单 │  │ 循环展开 │  │ 实时日志 │  │
│  └──────────┘  └──────────┘  └──────────┘  │
├────────────────────────────────────────────┤
│  WebSocket 通信层                           │
├────────────────────────────────────────────┤
│  后端执行引擎                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ 模板解析 │  │ 调度器   │  │ Worker池 │  │
│  │ Schema   │→│ 拓扑排序 │→│ 隔离执行 │  │
│  │ 校验     │  │ 流式推送 │  │ 资源限制 │  │
│  └──────────┘  └──────────┘  └──────────┘  │
├────────────────────────────────────────────┤
│  持久化层                                   │
│  SQLite（模板 + 执行历史）                  │
└────────────────────────────────────────────┘

部署方式: Docker Compose (studio-frontend + studio-backend)
外部依赖: 无（执行引擎不依赖 AI API）
```

## 命名体系

| 日常用名 | 内部代码名 | 定义 |
|---------|-----------|------|
| 步骤 | Node | 最小功能单元，一张卡片 |
| 模板 | Template | 完整工具，参数表单入口 |
| 子步骤 | Subflow | 循环内部的步骤组 |
| — | Wire | 步骤之间的数据连线（高级视图可见） |
| 循环步骤 | Loop | 可展开的特殊步骤 |
| 条件步骤 | Branch | 可分支的特殊步骤（暂不可用） |
| 后端引擎 | Engine | 运行在后端的执行器 |

**日常用语中避免出现**：节点、连线、子流程、作用域。这些是开发者/代码级用语，用户不需要接触。

## 数据流

```
模板参数（用户填）
    │
    ▼
  $param.xxx 引用  ────────→  步骤的 config 中
    │
    ▼
步骤执行（顺序 = steps 列表顺序）
    │
    ▼
  $steps.步骤ID 引用 ←─────── 上一步的输出被命名
    │
    ▼
loop 内部：子步骤按相同规则执行
    │
    ▼
最终步骤输出 → 展示给用户
```

### 变量引用语法

| 写法 | 含义 | 示例 |
|------|------|------|
| `$param.input_text` | 引用用户填的参数 | config.text: "$param.input_text" |
| `$steps.read_input` | 引用某步骤的输出 | config.source: "$steps.read_input" |
| `"prefix-$param.x-suffix"` | 内联拼接 | config.label: "处理-$param.name-完成" |
| `$$` | 字面量美元符号 | config.price: "$$19.99" |

**不支持：** 表达式求值、运算拼接、函数调用。变量引用只做字典查找。

## 安全边界

| 边界 | 说明 |
|------|------|
| 模板定义 | 声明式 YAML，不含可执行代码 |
| 模块白名单 | 动态加载的模块需在注册表中 |
| Worker 隔离 | 每步在独立 worker_threads 中执行 |
| 循环保护 | 模板上限 ≤ 10000 + 引擎环境变量兜底 |
| 数据不持久 | 执行结果存 SQLite（可选），不在外部传输 |

## 与旧 dtool 的关系

**旧 dtool（公共项目）**
- 纯前端 SPA，浏览器内执行
- 面向公网用户
- 进入长期维护模式，不做新功能

**dtool Studio（新项目）**
- 前端 + 后端架构
- 仅限个人/团队内部使用
- 活跃开发中

**双向反哺：** Studio 产生的纯函数模块可以移植到旧 dtool。UI 和交互设计理念也可以参考。但引擎、架构层面的能力不能反哺（架构不同）。
