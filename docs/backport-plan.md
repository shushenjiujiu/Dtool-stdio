# dtool Studio → 旧 dtool 反哺计划（草案）

> 维护: Manager (系统运维与安全审计官)
> 日期: 2026-06-26
> 状态: 草案 — 待 Heye 确认方向后细化

---

## 反哺原则

1. **新项目优先** — 第一阶段不反哺，全力推进 dtool Studio MVP
2. **自然流入** — Studio 成熟后，成熟的模块/设计模式自然移植
3. **保持兼容** — 旧 dtool 不引入 Studio 特有的依赖（后端运行时、WebSocket）
4. **不强制同步** — 两个项目可以各自演化，只移植明显有价值的

---

## 第一阶段（Studio MVP 后可做的）

### 1.1 模块移植（高价值）

| 模块 | Studio 中的形态 | 旧 dtool 移植方式 |
|------|----------------|-------------------|
| 循环容器 | `loop` 模块（嵌套子步骤） | 已有 ✅ 但可复用 Studio 的迭代逻辑 |
| combine 合并 | `combine` 模块（多输入合并） | 已有 ✅ |
| 条件分支 | `branch` 模块（双路径） | 目前旧 dtool 没有，建议移植 |
| 输入模块 | Studio 用 `input` 声明参数 | 旧 dtool 用字段 UI，**无需移植** |
| 输出模块 | Studio 用 `output` 收集结果 | 旧 dtool 用独立输出区，**无需移植** |

**建议：** 第一阶段暂不移植。等 Studio 的 `branch` 模块稳定后，花 1 天移植到旧 dtool。

### 1.2 模板系统（中价值）

- Studio 的 YAML 模板格式比旧 dtool 的 DSL 更结构化
- 旧 dtool 可以加一个"导入模板 YAML"的入口
- **但**旧 dtool 是纯前端，模板导入后端校验不可用
- **建议：** 等到 Studio 的模板库积累 10 个以上时，再做手动转换 DSL 的脚本

---

## 第二阶段（Studio 稳定后）

### 2.1 执行引擎拆包（高价值）

如果 Studio 的执行引擎抽成独立 npm 包（`@dtool/engine`）：
- 旧 dtool 可以直接引用
- 统一模块注册表 + 校验逻辑
- **优势：** 两套 UI 共享一套执行逻辑，模块开发者只需写一次

但这需要工程层面重构——旧 dtool 的当前引擎是内嵌在 React 组件里的，拆包需要较大改动。建议第二阶段再评估收益是否值得。

### 2.2 UI 设计模式（低价值）

- Studio 的"步骤列表 → 展开→底层图"的分层设计
- 旧 dtool 的 UI 风格完全不同（窄管道式），强行对齐反而奇怪
- **建议：** 不强制移植 UI 模式。保持两个项目各自的视觉语言。

---

## 结论

| 阶段 | 可反哺内容 | 工作量 | 建议 |
|------|-----------|--------|------|
| 第一阶段 | `branch` 模块 | ~1d | 等 Studio 模块稳定后移植 |
| 第一阶段 | YAML 模板导入 | ~2d | 等模板库 10+ 时再考虑 |
| 第二阶段 | 共享引擎包 | ~5d | 需评估收益 vs 重构成本 |
| 第二阶段 | UI 设计模式 | — | 不建议移植 |

**现阶段建议：** 先不做反哺。Studio MVP + 旧 dtool 维护并行，等 Studio 跑起来后再评估哪些真的值得移植。
